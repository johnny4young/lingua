# Windows signing

This guide covers direct-distribution Windows builds for Lingua. It does not
change the repository license; it prepares signed Windows artifacts before a
public Windows launch.

> For how these signed artifacts fit the end-to-end update signature chain
> (manifest → installer → on-disk binary) and the release-time gates that
> enforce it, see
> [`RELEASE_SECURITY.md` § Signature chain](./RELEASE_SECURITY.md#signature-chain).

## Prerequisites

- A Windows code-signing strategy for Authenticode signatures.
- Access to a `windows-latest` GitHub Actions runner.
- GitHub Actions secrets configured for the chosen strategy.
- Lingua's current Windows path from `forge.config.ts`: Squirrel.Windows
  installers generated by `@electron-forge/maker-squirrel`.

Official references:

- [Electron Forge Windows signing](https://www.electronforge.io/guides/code-signing/code-signing-windows)
- [Electron Forge Squirrel.Windows maker](https://www.electronforge.io/config/makers/squirrel.windows)
- [Microsoft code-signing options for Windows apps](https://learn.microsoft.com/en-us/windows/apps/package-and-deploy/code-signing-options)
- [Microsoft Artifact Signing](https://azure.microsoft.com/en-us/products/trusted-signing)
- [Azure Trusted Signing GitHub Action](https://github.com/Azure/trusted-signing-action)

## Current Lingua CI path

The active release workflow supports the PFX-backed Authenticode path:

| Secret | Purpose |
|--------|---------|
| `WIN_CERT_FILE` | Base64-encoded `.pfx` certificate content. |
| `WIN_CERT_PASSWORD` | Password for the `.pfx` certificate. |

`.github/workflows/release.yml` fails fast when either secret is missing,
materializes `WIN_CERT_FILE` into `$RUNNER_TEMP/lingua-signing.pfx`, runs
`pnpm run make:desktop:win`, and verifies the produced executable with
`Get-AuthenticodeSignature`.

`forge.config.ts` passes the certificate through Electron Forge's
`windowsSign` configuration so Squirrel signs the generated Windows payload.

## Create or obtain a signing certificate

Do not use a self-signed certificate for public distribution.

For the current CI path, use a certificate authority flow that can produce a
PFX suitable for CI signing, or a provider-managed export that is explicitly
allowed by the certificate terms. Since many modern code-signing certificates
keep private keys in hardware or managed HSM services, confirm this with the
provider before buying.

If the provider requires hardware-token, HSM, or cloud-managed signing instead
of an exportable PFX, do not force it into `WIN_CERT_FILE`. Use a follow-up
release-infra slice to wire `@electron/windows-sign` through provider-specific
`signWithParams`, `signToolPath`, or a signing hook, or sign the produced
artifacts after `make:desktop:win` using Azure Artifact Signing / DigiCert
KeyLocker / another supported HSM-backed service.

## Encode the PFX for GitHub Actions

From macOS:

```bash
base64 -i LinguaCodeSigning.pfx | pbcopy
```

From PowerShell:

```powershell
[Convert]::ToBase64String([IO.File]::ReadAllBytes("LinguaCodeSigning.pfx")) | Set-Clipboard
```

Set the copied value as `WIN_CERT_FILE` and the PFX password as
`WIN_CERT_PASSWORD` in GitHub Actions secrets. Never commit the `.pfx`, the
base64 output, or the password.

## Local and CI validation

Windows signing must be validated on Windows because the Forge/Squirrel path
uses SignTool-compatible Authenticode signing.

Fast CI preflight:

1. Configure `WIN_CERT_FILE` and `WIN_CERT_PASSWORD` in GitHub Actions.
2. Run the `Release` workflow with `release_windows` enabled and the other
   desktop platforms disabled if you only want to test Windows.
3. Keep the created release in draft.
4. Confirm the workflow summary includes `Windows signing` with
   `Verification: Valid`.

Local Windows verification after a build:

```powershell
pnpm run make:desktop:win
Get-ChildItem out/make -Recurse -Filter *.exe |
  ForEach-Object { Get-AuthenticodeSignature $_.FullName }
```

The release is not promotable until every installer executable reports
`Status: Valid`.

## Rotation

If the PFX, base64 value, password, provider account, or GitHub secret is
exposed:

1. Revoke or replace the affected certificate or signing profile with the
   certificate authority/provider.
2. Generate fresh signing material.
3. Replace the GitHub Actions secrets.
4. Re-run the Windows release workflow and verify `Get-AuthenticodeSignature`
   before promoting a draft release.
