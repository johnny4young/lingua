# Accessibility

Lingua ships with a tiered accessibility quality gate: automated WCAG
2.1 AA scans for renderer overlays, keyboard-only flow tests, focus
restoration tests, and a manual screen-reader checklist for surfaces
that automation cannot reliably cover.

## Automation coverage

The gate runs as part of `pnpm run test:e2e:web` via `tests/e2e/a11y.spec.ts`.

WCAG tag set: `wcag2a`, `wcag2aa`, `wcag21a`, `wcag21aa`. Findings
classified as `serious` or `critical` fail the build. `moderate` and
`minor` findings surface in test output for follow-up without
blocking.

The Spanish locale is re-scanned for Settings → General and the
Command Palette so locale-specific aria-label regressions get
caught. Compact viewport (800x600) is exercised once for the
baseline shell and the Settings overlay.

Developer Utilities now lives in an editor workspace tab instead of a
modal; its accessibility contract is keyboard navigation inside the
search/list/workspace surface plus a closable editor tab, not modal
focus restoration.

StatusBar segments remain keyboard-focusable controls per internal Their
informational content now exposes `role="status"`: lint, offline, and run
updates use polite live announcements, while cursor, encoding, and Git
retain status semantics with live announcements disabled to avoid noisy
cursor-by-cursor screen-reader output.

All user-facing numeric counters and byte sizes use the selected Lingua
locale rather than the host browser locale. The E2E gate verifies the
same Settings value as `10,000` in English and `10.000` in Spanish.

## Excluded surfaces and silenced rules

`tests/e2e/a11y.helpers.ts` excludes two upstream surfaces from every
scan and silences one rule with a documented tooling rationale:

- `.monaco-editor` — Monaco ships its own a11y model; we cannot fix
  upstream issues from this repo. Monaco itself supports an
  `accessibilitySupport: 'on'` mode that screen-reader users should
  enable in their editor settings.
- `.vim-status-bar` — rendered by `monaco-vim` and inherits Monaco's
  a11y posture.
- Rule `color-contrast` — silenced because **axe-core 4.11 does not
  fully resolve `oklch()` color values**, and the entire Lingua DS
  token system is expressed in `oklch()`. axe falls back to incorrect
  RGB values and reports false-positive contrast failures against
  tokens the browser actually renders with plenty of contrast. This
  is verified empirically: `getComputedStyle(...).color` on the
  `.leading-7` description paragraph returns
  `oklch(0.72 0.018 210)` (light gray) on a `oklch(0.12 0.005 210)`
  background — well above WCAG 2.1 AA — yet axe reports the same
  foreground as `#21292a` at 1.31:1. The rule is re-enabled for any
  manual run via `auditA11y(page, { disabledRules: [] })`. Track
  upstream axe-core for OKLCH support; drop the silence when it
  lands.

Manual contrast checks (the `Color Contrast Analyser` GUI on
macOS/Windows, or DevTools' built-in inspector) cover the
real contrast story for the dark and light shells. Both shells
were verified at WCAG 2.1 AA for body text + UI controls during
internal implementation.

## Manual screen-reader checklist

Automation cannot fully validate Voice / NVDA announcements. Run
this short manual pass before any release that touches Settings,
the toolbar, or a new overlay/workspace.

### macOS / VoiceOver

Activate VoiceOver: `Cmd+F5`.

1. **Toolbar tour.** Tab through the toolbar from left to right.
   Every icon button should announce its label (e.g. "Toggle sidebar
   (Cmd+B), button"). The Run button announces its enabled / disabled
   state.
2. **Settings open.** Press `Cmd+,`. VoiceOver announces "dialog,
   Tune the shell, editor, and runtime defaults". Tab into the
   tablist and arrow through the tabs — each tab announces its
   label and selected state.
3. **Settings toggles.** Find a toggle (e.g. Editor → Show line
   numbers). VoiceOver announces "switch, on" or "switch, off". The
   visible label is read first thanks to `aria-labelledby` from the
   parent Row.
4. **Status notice.** Trigger a status notice (e.g. paste an invalid
   license token). VoiceOver should announce the localized message
   automatically because the notice container declares `role="status"`
   and `aria-live="polite"`.
5. **StatusBar.** With the StatusBar enabled, run code and introduce a
   lint error. VoiceOver should announce those state changes without
   repeating cursor position on every movement.
6. **Settings dismiss.** Press `Escape`. Focus returns to the toolbar
   Settings button — VoiceOver re-announces it.

### Windows / NVDA

Activate NVDA: `Ctrl+Alt+N`.

1. Same toolbar / Settings / dismiss flow as the macOS pass.
2. Pay special attention to the Command Palette result list. NVDA
   should announce option count and current selection as you arrow
   through. (Automation covers visibility but not announce text.)

## Known limitations

- **Monaco editor surface.** Code editing inside Monaco follows
  Monaco's own a11y model. Power users should enable Monaco's
  `accessibilitySupport: 'on'` mode via the editor command palette.
- **Vim status bar.** Inherits Monaco's posture. Vim mode users who
  rely on screen readers should validate their flow against Monaco's
  upstream documentation.
- **Color-contrast comments.** The dark-first DS comment color is
  intentionally below AA. We silenced the axe rule pending a DS
  decision on whether to bump the token or add a separate
  high-contrast theme pack. Until then, users sensitive to low
  contrast should switch to the Lingua Light theme via Settings →
  Appearance.
- **Rich Developer Utilities output.** Some panels render visual
  output (diff viewer, color tools, timestamp grids) that screen
  readers cannot announce in a useful way. The input controls are
  fully labeled; the output panels are scoped to the visual track.

## Reporting a regression

If you hit an a11y regression in a current release:

1. Confirm the surface is listed in the **Automation coverage** table
   above. If it is, run the suite locally:
   `pnpm run test:e2e:web -- --grep "a11y.spec"`. A passing local run
   suggests the regression is environmental.
2. Open a GitHub issue against the repo with the `accessibility`
   label. Include the surface, the screen-reader / browser, and a
   reproduction trace if possible.

## Why a tiered gate

Hard automation alone misses what a screen reader actually hears.
Manual passes alone are skipped under deadline pressure. The tiered
shape — automated for regression-proof patterns, manual for
announce-text and complex flows — keeps the floor high without
blocking shipping on a perfect run.
